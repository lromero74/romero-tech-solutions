## 1. Problem statement

The agent already detects and reports outdated packages from every package manager it can find — apt/dnf/pacman/zypper, pip, npm, gem, cargo, snap, flatpak, brew, winget, choco — and surfaces them on the dashboard's *Asset Inventory* tab (`src/components/admin/agent-details/AssetInventory.tsx`, schema in `backend/migrations/036_add_package_manager_tracking.sql`). But there is **no way from the dashboard to actually apply any of those updates.** Today the only path is for an admin to SSH into the device and run the right command per package manager. For an MSP operating across hundreds of devices that's a non-starter — it's the "remote patch management" gap that every commercial RMM (Datto, NinjaOne, Atera, Action1) fills natively.

This PRP scopes the agent + backend + frontend changes to close that gap.

---

## 2. Goals (in scope)

1. **Apply updates from the dashboard.** A privileged dashboard user can click a button next to an outdated package (or "Update all") on a device's inventory page, and the device's agent applies the update.
2. **Per-package selection.** Either all outdated packages in a manager, or a hand-picked subset.
3. **Per-manager dispatch.** apt, dnf, pacman, zypper, brew, pip, pip3, npm (global), gem, cargo, snap, flatpak, winget, choco. Manager-specific best-practice flags (no-prompt, accept-keys, no-recommends-toggle, etc.).
4. **Structured results.** Per-package outcome (`succeeded` / `failed` / `skipped`), captured stdout/stderr, exit code, before/after version. Persisted as an audit record on the backend, viewable in the UI.
5. **Authorization & audit.** Only users with a new `agent:execute_updates` permission can trigger updates. Every action recorded in `agent_command_audit_log` with actor, target device, payload, result, and IP address.
6. **Status feedback in the UI.** While the update is running, the row shows a spinner; on completion the row updates to the new state (gone, or moved to "still outdated" if the update failed).
7. **Scheduling hook (basic).** Optionally specify a maintenance window (e.g. "run after 2026-05-01 02:00 UTC"). Backed by the existing `policySchedulerService`.

## 3. Non-goals (out of scope for v1)

- **Reboot orchestration.** Some package updates require reboot (kernel, glibc, etc.). v1 reports `reboot_required: true` in the result; physically rebooting is a separate command (we can add `reboot_device` later).
- **Cross-device fleet rollouts.** v1 is per-device. "Update all 200 Linux servers" wave-rollouts come in v2 — the same command structure plus a fleet orchestrator.
- **Custom repo selection or pinning.** v1 uses the device's currently-configured repos. Switching repos / pinning versions is a separate feature.
- **Rollback.** Most package managers don't support clean rollback anyway. v1 records the before/after versions and the user can manually re-pin if needed.
- **Software install (as opposed to update).** Same code path is reusable later, but v1 only covers `<package> X → version Y where Y > X`.

## 4. User stories

> As an MSP technician I see that `glibc` on a customer's Fedora server has 3 critical security advisories pending. I click *Update glibc* on the dashboard. The agent applies the update via `dnf upgrade -y glibc`, reports success and the new version, and the inventory page refreshes to show glibc as up to date. The audit log records that I (`alice@msp.com`) triggered it at 2026-05-12 09:14 UTC.

> As an MSP technician I look at a Mac and see Homebrew has 12 outdated formulae. I click *Update all (Homebrew)*. Each one reports back individually; one (`postgresql@14`) fails because of a tap conflict. The dashboard shows 11 ✓ and 1 ✗ with the failing brew's stderr available on click.

> As an MSP owner I require that only my senior techs can apply Linux kernel updates. I configure a permission policy: `agent:execute_updates:linux:kernel` requires role `senior_tech`. Junior techs can apply non-kernel updates but see the kernel row as read-only.

```
                                                 ┌────────────────────────┐
   ┌─────────────────────┐    POST update_request │   RomeroTechSolutions  │
   │  Dashboard frontend │ ──────────────────────▶│       backend (API)    │
   │  AssetInventory.tsx │                        │  routes/agents.js etc. │
   └─────────────────────┘ ◀────────────────────  │                        │
              ▲              GET command_status   │  ┌──────────────────┐  │
              │                                   │  │ agent_commands   │  │
              │ websocket push                    │  │   table          │  │
              │ (agent_command_completed)         │  └──────────────────┘  │
              │                                   └────────┬───────────────┘
              │                                            │
              │                                            ▼ poll (existing)
   ┌──────────┴────────────┐                      ┌────────────────────────┐
   │  rts-monitoring-agent │ ◀──poll cmds──────── │   /agents/:id/commands │
   │  internal/commands/   │ ──submit results──▶  │   /agents/:id/results  │
   │  + new updates pkg    │                      │                        │
   └───────────────────────┘                      └────────────────────────┘
              │
              ▼ exec (sudo / pkexec / native)
   ┌─────────────────────────────────────────────┐
   │   Local package manager (dnf / apt / etc.)   │
   └─────────────────────────────────────────────┘
```

**Wire format**: extends the existing `Command` shape in `internal/commands/commands.go` with a new `command_type = "update_packages"`. No new transport — same poll/result loop already used by `ping`/`run_script`/`restart_agent`.

## 6. Detailed design

### 6.1 Agent side (`rts-monitoring-agent`)

#### 6.1.1 New package: `internal/updatepkgs/`

```
internal/updatepkgs/
├── updates.go         // Service entry, dispatch table
├── managers/
│   ├── apt.go         // apt-get update + apt-get install --only-upgrade
│   ├── dnf.go         // dnf upgrade
│   ├── pacman.go      // pacman -Syu / -S
│   ├── zypper.go      // zypper update
│   ├── brew.go        // brew upgrade
│   ├── pip.go         // pip install --upgrade
│   ├── npm.go         // npm install -g pkg@latest
│   ├── gem.go         // gem update
│   ├── cargo.go       // cargo install-update (requires cargo-update)
│   ├── snap.go        // snap refresh
│   ├── flatpak.go     // flatpak update
│   ├── winget.go      // winget upgrade
│   └── choco.go       // choco upgrade
├── results.go         // Structured PerPackageResult with marshaling
└── safety.go          // Pre-flight checks (sudo available, network, etc.)
```

#### 6.1.2 Command payload (server → agent)

```json
{
  "id": "cmd_01HXYZ...",
  "command_type": "update_packages",
  "payload": {
    "manager": "dnf",
    "scope": "selected",                    // "all" | "selected" | "security_only"
    "packages": ["glibc", "openssl"],       // ignored when scope="all"|"security_only"
    "dry_run": false,                       // if true, simulate only
    "auto_confirm": true,                   // pass --yes / -y / --noconfirm
    "timeout_seconds": 1800,                // hard cap per package manager invocation
    "stop_on_first_failure": false          // if true, abort the rest after first ✗
  }
}
```

#### 6.1.3 Result payload (agent → server)

```json
{
  "command_id": "cmd_01HXYZ...",
  "status": "completed",                  // "completed" | "completed_with_failures" | "failed"
  "started_at": "2026-05-12T09:14:01Z",
  "ended_at":   "2026-05-12T09:14:42Z",
  "manager": "dnf",
  "manager_version": "4.20.0",
  "results": [
    {
      "package": "glibc",
      "outcome": "succeeded",             // "succeeded" | "failed" | "skipped"
      "version_before": "2.39-12.fc43",
      "version_after":  "2.39-13.fc43",
      "stdout": "…",
      "stderr": "",
      "exit_code": 0,
      "duration_ms": 18420,
      "reboot_required": true,
      "skipped_reason": null              // populated when outcome="skipped"
    },
    { "package": "openssl", "outcome": "succeeded", ... }
  ],
  "summary": {
    "total":     2,
    "succeeded": 2,
    "failed":    0,
    "skipped":   0,
    "reboot_required": true               // OR over per-package rebootRequired
  }
}
```

#### 6.1.4 Privilege model

| OS | Approach |
| --- | --- |
| **Linux** | The system daemon already runs as **root** (systemd `rts-agent.service`). It receives the command and executes the package manager directly — no `sudo`/`pkexec` needed. |
| **macOS** | Daemon runs as root via LaunchDaemon. Same as Linux for system managers (e.g. `softwareupdate`). For Homebrew (which is per-user), the agent shells out via `sudo -u <brew-owner>` — owner discoverable via `stat -f '%Su' $(brew --prefix)`. |
| **Windows** | Daemon runs as `LocalSystem`. winget/choco invoked directly. |

The agent does NOT use `pkexec` or any GUI privilege escalation for this path — those are user-driven; this is automated and must run head­less.

#### 6.1.5 Command dispatch

Add to `internal/commands/commands.go:executeCommand`:
```go
case "update_packages":
    return s.executeUpdatePackages(cmd)
```

`executeUpdatePackages` unmarshals the payload, looks up the manager in a per-OS dispatch table, and calls the manager's `Apply(ctx, payload) ([]PerPackageResult, error)`.

#### 6.1.6 Per-manager command construction

| Manager | "selected" cmd | "all" cmd | "security_only" cmd |
| --- | --- | --- | --- |
| apt | `apt-get install --only-upgrade -y <pkg>...` | `apt-get upgrade -y` | `unattended-upgrade --dry-run` then targeted (apt has no native scope; we filter via `apt-mark showhold` + `apt list --upgradable -a`) |
| dnf | `dnf upgrade -y <pkg>...` | `dnf upgrade -y` | `dnf upgrade --security -y` |
| pacman | `pacman -S --noconfirm <pkg>...` | `pacman -Syu --noconfirm` | n/a (no native security scope) |
| zypper | `zypper -n install --only-update <pkg>...` | `zypper -n update` | `zypper -n patch --category security` |
| brew | `brew upgrade <pkg>...` | `brew upgrade` | n/a |
| pip / pip3 | `pip install --upgrade <pkg>...` | iterate over `pip list --outdated --format=json` | n/a |
| npm (global) | `npm install -g <pkg>@latest` | iterate over `npm outdated -g --json` | n/a |
| gem | `gem update <pkg>` | `gem update` | n/a |
| cargo | `cargo install-update <pkg>` (req: `cargo install-update`) | `cargo install-update --all` | n/a |
| snap | `snap refresh <pkg>` | `snap refresh` | n/a |
| flatpak | `flatpak update -y <ref>...` | `flatpak update -y` | n/a |
| winget | `winget upgrade --silent --id <id>` | `winget upgrade --all --silent` | n/a |
| choco | `choco upgrade -y <pkg>` | `choco upgrade all -y` | n/a |

For each, the manager module returns a `PerPackageResult` per requested package by parsing the manager's machine-readable output (`-oDpkg::Use-Pty=0 ... | apt-get`, `dnf -q ... --json`, etc.) where available, and falling back to wrapping each package in its own invocation when the manager lacks a structured output mode.

#### 6.1.7 Safety pre-flight (`safety.go`)

Before executing, the agent verifies:
1. The named manager binary exists.
2. Network is reachable (so we don't waste time on a manager that needs a repo refresh).
3. Disk has > 1 GB free in the manager's cache dir.
4. No other rts-agent update job is already running for the same manager (file lock at `/var/lib/rts-agent/locks/update_<manager>.lock`).
5. For Linux daemons: `systemctl is-active dbus` — some package managers fail oddly in a session-less context without it.
6. (Windows) UAC manifest check.

If any pre-flight fails, the command returns `status="failed"` with a structured `pre_flight_error`.

#### 6.1.8 Logging

Every update operation emits structured log lines (already journal'd on Linux):

```
2026/05/12 09:14:01 🛠️  update_packages: dnf, scope=selected, packages=[glibc openssl]
2026/05/12 09:14:01 🛠️    + dnf upgrade -y glibc
2026/05/12 09:14:42 🛠️    ✓ glibc 2.39-12.fc43 → 2.39-13.fc43 (18.4s)
...
```

This means *View Logs* in the tray (the journalctl-backed viewer we just shipped) shows the full update trail in real time.

### 6.2 Backend side (`RomeroTechSolutions`)

#### 6.2.1 New tables

`backend/migrations/0NN_agent_command_audit.sql`:

```sql
CREATE TABLE agent_commands (
  id                UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  agent_id          UUID         NOT NULL REFERENCES agents(id),
  requested_by      UUID         NOT NULL REFERENCES users(id),
  command_type      TEXT         NOT NULL,           -- e.g. 'update_packages'
  payload           JSONB        NOT NULL,
  status            TEXT         NOT NULL DEFAULT 'pending',
                                                      -- pending|in_progress|completed|completed_with_failures|failed|expired
  scheduled_for     TIMESTAMPTZ,                      -- NULL = immediate
  started_at        TIMESTAMPTZ,
  ended_at          TIMESTAMPTZ,
  result            JSONB,                            -- the agent's full result payload
  ip_address        INET,                             -- requester's IP
  user_agent        TEXT,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
CREATE INDEX agent_commands_agent_status_idx ON agent_commands (agent_id, status);
CREATE INDEX agent_commands_requested_by_idx ON agent_commands (requested_by, created_at DESC);
```

#### 6.2.2 New routes (`backend/routes/agents.js`)

```
POST   /api/agents/:agentId/commands/update_packages
       body: { manager, scope, packages, dry_run, schedule_for, ... }
       → returns { command_id, status: "pending", ... }
       (stores in agent_commands; the agent's existing /commands poll picks it up)

GET    /api/agents/:agentId/commands/:commandId
       → returns the full record incl. result if completed

GET    /api/agents/:agentId/commands?type=update_packages&limit=50
       → list of recent commands for this device, for the audit panel

POST   /api/agents/:agentId/commands/:commandId/cancel
       → if status=pending, marks expired so the agent skips it on next poll

(internal, agent-facing — already exists)
GET    /api/agents/:agentId/commands       (poll, the /commands endpoint)
POST   /api/agents/:agentId/commands/:commandId/result
       (agent submits the structured result here; backend updates row + emits ws event)
```

#### 6.2.3 Authorization

New permission (added via the existing RBAC system, see `migrations/054_comprehensive_rbac_enhancements.sql` for the precedent):

```
agent:execute_updates              # baseline: can trigger any package update
agent:execute_updates:kernel       # extra gate for kernel-class updates
                                    # (auto-applied when manager=dnf|apt and the package
                                    #  matches /^(linux-|kernel-|kernel$)/)
```

Default mapping:
- `client_admin` role → `agent:execute_updates`
- `senior_technician` role → `agent:execute_updates`, `agent:execute_updates:kernel`
- `technician` role → `agent:execute_updates`
- `viewer` role → no command permissions at all

The route handler enforces the permission, plus a per-device check that the agent currently belongs to the requester's organization.

#### 6.2.4 Audit emission

Every command insert + every status transition logs to the existing `audit_logs` table with `event_type='agent_command'` so the security review already in place picks it up automatically.

#### 6.2.5 Websocket push

When the agent submits a result and the backend updates the row, emit `agent.command.completed` over the existing socket.io channel (per-organization room). Frontend listens and updates the dashboard row in place — no polling.

#### 6.3.2 UX flow (single-package)

1. User on AssetInventory tab sees `glibc 2.39-12.fc43 → 2.39-13.fc43` in the outdated list, clicks **Update**.
2. Modal opens: confirms manager (`dnf`), package (`glibc`), the exact command preview (`dnf upgrade -y glibc`), schedule picker (default *Now*), risk warning (kernel class only), Confirm/Cancel.
3. User clicks **Confirm**. Frontend POSTs `/agents/:id/commands/update_packages`. Modal switches to "Queued — waiting for agent (typical 10–60s)…".
4. Agent's next poll (~30s) picks up the command, executes, posts result.
5. Backend pushes `agent.command.completed` over websocket. Frontend modal flips to *Completed* / *Failed*, shows per-package result table, refreshes the underlying inventory list.

#### 6.3.3 UX flow (Update all)

Same modal, but the package list is populated from the inventory snapshot. User can uncheck individual packages to opt out. Confirms with explicit count: "Update 17 packages with apt? [Cancel] [Update]".

## 7. Per-manager implementation matrix

(See agent-side table in §6.1.6 for the canonical command set. The implementation contract for each manager:)

```go
package managers

type PerPackageResult struct {
    Package        string  `json:"package"`
    Outcome        string  `json:"outcome"`        // succeeded|failed|skipped
    VersionBefore  string  `json:"version_before"`
    VersionAfter   string  `json:"version_after"`
    Stdout         string  `json:"stdout"`
    Stderr         string  `json:"stderr"`
    ExitCode       int     `json:"exit_code"`
    DurationMS     int64   `json:"duration_ms"`
    RebootRequired bool    `json:"reboot_required"`
    SkippedReason  string  `json:"skipped_reason,omitempty"`
}

type Manager interface {
    Name() string
    Detect() bool                                           // is this manager available on this host?
    SnapshotVersion(pkg string) (string, error)             // for before/after capture
    Apply(ctx context.Context, payload UpdatePayload) ([]PerPackageResult, error)
    RequiresReboot() ([]string, error)                       // post-update check (e.g. /usr/bin/needs-restarting on dnf)
}
```

Reboot detection per OS:
- **Debian/Ubuntu** — `[ -f /var/run/reboot-required ]`
- **RHEL/Fedora** — `dnf needs-restarting -r` (exit 1 = reboot needed)
- **Arch** — diff `pacman -Q linux*` with `uname -r`
- **macOS** — `softwareupdate --list` returns "Action: restart"
- **Windows** — registry key `HKLM\Software\Microsoft\Windows\CurrentVersion\Component Based Servicing\RebootPending`

## 8. Security & authorization policy

1. **Default deny.** Without `agent:execute_updates`, the route returns 403 with no command-list visible.
2. **Org isolation.** Backend joins `agents.organization_id` against the requester's session — no cross-org targeting.
3. **Kernel-class extra gate.** As above, kernel/glibc-tier updates require `:kernel` permission.
4. **Replay protection.** Each command has a server-issued UUID; agent rejects a command it already has stored in `~/.cache/rts-agent/recent-commands.json` (24-hour window).
5. **TLS only.** Same auth chain as everything else (mTLS + JWT bearer).
6. **Audit all.** Every command + every result write logs to `audit_logs`. Logs are immutable, retained 1 year minimum, surfaced in the security audit dashboard.
7. **Rate limit.** Max 100 update commands per organization per hour at the API gateway. Stops a compromised admin token from kicking off a fleet-wide patching event.
8. **Maintenance window honoring.** If the agent has a maintenance-window policy and the command is sent outside it without `schedule_for` set, the agent NACKs the command with `pre_flight_error: "outside_maintenance_window"`. The backend re-queues it for the next available window.

## 10. Open questions

1. **Should the agent refuse `package=*` (or scope=all) for managers without idempotent dry-run?** Probably yes — for first release, require explicit per-package selection on those.
2. **How do we surface partial failures?** Agreed: `status="completed_with_failures"` plus per-package detail. Frontend shows yellow not red.
3. **Locale for stdout/stderr capture.** Force `LC_ALL=C` so error parsing is reliable across locales — even though it makes the user-visible output English-only.
4. **Snap's "running" check.** Snap refuses to refresh apps that are currently running. Should the agent kill them? **No.** Skip with `outcome=skipped`, `skipped_reason="snap_app_running"`. Tech can retry off-hours.
5. **Brew on multi-user macs.** If two humans both have Homebrew-prefix-owned, which one's brew do we run? Pick the prefix that owns the brewfile of the package. Document explicitly.
6. **Windows Store apps via winget.** Some require user interaction. Skip them: `winget upgrade --silent --include-unknown --include-pinned` produces a deterministic set; skip the rest.
7. **Should "kernel" auto-trigger a reboot?** No. Always report `reboot_required: true` and let the human or a separate explicit `reboot_device` command do it.

## 11. Risks

| Risk | Mitigation |
| --- | --- |
| Bad update bricks a device | Per-manager dry-run mode for confirm-step preview; pre-flight backup hook (out of scope v1 but called out); keep audit trail |
| Compromised admin token kicks off fleet update | Rate limit + audit log + alert thresholds (§8.7) |
| Manager command output format drifts across versions | Capture the manager version in the result; integration tests pin exact output assertions to specific versions; fallback to "best effort" parsing if version unknown |
| Locale/encoding makes parser fail | `LC_ALL=C` (§10.3) |
| Agent locks itself running concurrent updates | File-locks (§6.1.7 #4); job queue serializes per-manager |
| Network drop mid-update | `apt`/`dnf` are transactional in the dpkg/rpm db; mid-update kill leaves the txn consistent. Snap and flatpak similarly journaled. Pacman has known issues — document and warn |

## 12. Success criteria

- A tech can apply at least one outdated package update from the dashboard end-to-end on a Fedora, an Ubuntu, and a macOS device.
- Audit log records every action with actor + result.
- 99th-percentile end-to-end latency (dashboard click → result visible) under 90 seconds for a single small package.
- Failure rate < 2% on the test fleet over 1 week of dogfooding.
- No security review finding above "Low" severity on the new endpoints.
